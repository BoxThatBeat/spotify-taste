<task type="checkpoint:human-verify" gate="blocking">
  <what-built>
Complete two-user sequential OAuth authorization system with:
- Express server with session management
- OAuth routes for User A and User B (/login/userA, /login/userB, /callback)
- Landing page with authorization buttons and status display
- CSRF protection via state parameter validation
- Duplicate account detection
- Session timeout (15 minutes)
- Server-side token storage (tokens never exposed to frontend)
  </what-built>
  
  <how-to-verify>
**Prerequisites:**
1. Ensure .env file exists with valid Spotify credentials (CLIENT_ID, CLIENT_SECRET, REDIRECT_URI)
2. Verify Redirect URI (http://localhost:3000/callback) is configured in Spotify Developer Dashboard
3. Start server: `npm start` or `node server.js`
4. Server should log "Server running on http://localhost:3000"

**Test Scenarios:**

**Scenario 1: Happy Path (Both Users Authorize Successfully)**
1. Open http://localhost:3000 in browser
2. Click "Connect Your Spotify" button
3. Verify: Redirects to Spotify authorization page
4. Authorize on Spotify (use Test Account A)
5. Verify: Redirects back to localhost, shows "✓ Connected as [username]"
6. Verify: "Connect Friend's Spotify" button appears
7. Click "Connect Friend's Spotify"
8. Verify: Redirects to Spotify authorization page
9. Authorize on Spotify (use DIFFERENT Test Account B)
10. Verify: Redirects back, shows both users connected with checkmarks

**Scenario 2: Duplicate Account Detection**
1. Clear session: POST to http://localhost:3000/api/clear-session (use curl or browser console)
2. Authorize User A with Account A
3. Click "Connect Friend's Spotify"
4. Authorize User B with SAME Account A
5. Verify: Error message displayed: "This account already authorized. User B needs a different account."
6. Verify: Can retry with different account

**Scenario 3: Authorization Cancellation**
1. Clear session
2. Click "Connect Your Spotify"
3. On Spotify authorization page, click "Cancel" or close tab
4. Verify: Returns to landing page or shows error
5. Verify: Can retry authorization from beginning

**Scenario 4: Session Persistence**
1. Complete full authorization (both users)
2. Refresh the page (F5)
3. Verify: Both users still show as connected (session persists)
4. Verify: No need to re-authorize

**Scenario 5: Session Timeout** (Optional - takes 15 minutes)
1. Authorize User A
2. Wait 15+ minutes without activity
3. Click "Connect Friend's Spotify"
4. Verify: Session expired, need to restart from beginning

**Scenario 6: Direct URL Access**
1. Clear session
2. Directly visit http://localhost:3000/login/userB (without authorizing User A first)
3. Verify: Returns error "User A must authorize first"

**Security Verification:**
1. Open browser DevTools → Network tab
2. Complete authorization flow
3. Verify: No access_token or refresh_token visible in browser (check Network requests, LocalStorage, Cookies)
4. Verify: Only session cookie exists (should be httpOnly, not readable by JavaScript)
5. Verify: OAuth state parameter is different each time authorization starts

**Expected Outcomes:**
- ✓ Both users can authorize successfully with different accounts
- ✓ Duplicate account is blocked with friendly error message
- ✓ Session persists across page refreshes
- ✓ Tokens stored server-side only (not visible in browser)
- ✓ State parameter provides CSRF protection
- ✓ User B cannot authorize before User A
  </how-to-verify>

<task type="auto">
  <name>Task 1: Create comparison logic and API endpoint</name>
  <files>routes/comparison.js</files>
  <action>
Create `routes/comparison.js` with POST /api/compare endpoint implementing privacy-preserving comparison:

**Comparison logic (use native JavaScript Set for O(1) lookup):**
1. Extract artist IDs from both users' data into Sets
2. Extract track IDs from both users' data into Sets
3. Calculate intersections using Set.filter() + Set.has() pattern (Node 20 doesn't have Set.intersection() yet)
4. Calculate Jaccard index for artists: |A ∩ B| / (|A| + |B| - |A ∩ B|) * 100
5. Calculate Jaccard index for tracks: same formula
6. Calculate overall match: combine all IDs (artists + tracks), then apply Jaccard formula
7. Enrich shared IDs with full metadata (artist name, images, track titles, album covers)

**API endpoint structure:**
- POST /api/compare with optional body: { timeRange: 'medium_term' }
- Validate both users are authorized (check req.session.tokens.userA and userB)
- Fetch both users' top artists and tracks using timeRange parameter
- Reuse token refresh pattern from spotify-data.js (check isTokenExpired, call refreshUserToken)
- Call comparison logic with fetched data
- Return JSON: { sharedArtists: [...], sharedTracks: [...], matchPercentage: X, breakdown: { artistsMatch: Y, tracksMatch: Z }, counts: { ... } }
- Handle errors: 400 if users not authorized, 500 on fetch/comparison failures

**AVOID:**
- Do NOT use Lodash or external libraries (native Set is optimal)
- Do NOT compare objects directly (compare IDs as strings)
- Do NOT divide by zero (check unionSize > 0 before calculating percentage)
- Do NOT send full user data to frontend (only shared items)
- Do NOT use nested loops (use Set.has() for O(1) lookup)

**Match research pattern from 03-RESEARCH.md:**
- Use exact code example from "Complete Comparison Function" section
- Follow "Privacy-Preserving Server-Side Comparison" pattern
- Implement all 5 common pitfall guards (empty sets, reference vs value, Jaccard formula, metadata enrichment, stale data)
